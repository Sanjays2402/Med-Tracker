## Per-feature notes

### regimen-snapshot-archive-history-rollup-csv-export

First CSV companion to the history rollup. Two CSVs (eventsCsv +
timelineCsv) because clinicians ask for different shapes:
- eventsCsv (rows of change events) is for sharing the timeline with
  a second-opinion specialist who imports it into Excel and sorts /
  filters in-place.
- timelineCsv (rows of snapshots with delta) is for plotting
  regimen-size-over-time in the recipient's analytics tool.

Two helpers split out the convenience cases: `exportRegimenHistoryEventsCsv`
(events alone) and `exportRegimenHistoryEventsCsvForMedication`
(single-medication slice). MED_TRACKER CSV conventions parallel
`dose-export-csv`: RFC 4180 quoting, optional BOM for Excel-on-Windows
UTF-8, header always emitted (even on empty rollup), empty cells for
null fields (NOT the literal string "null").

Critical product call: `eventOrder` defaults to 'medication' (rollup
order with chronological-inside-medication) because that's how the
prescriber READS the timeline (per-drug, change-by-change). The
alternative 'time' (flat chronological across meds) exists for
analytics tooling that wants the events stream as a fact table.

### dose-export-csv-import-roundtrip-validator-summary-text

Multi-line text companion to `summarizeRoundtripResult`'s one-liner.
Use case: the on-call engineer at 2am reading a CI artifact, or QA
tooling scanning a pipeline log, wants the per-tier sample doseIds
and parser-skip reason groupings — not just the headline.

Output is a fenced block (=== delimiters) suitable for cron logs, CI
artifacts, terminal stdout, or a Slack code-fenced message. Tier
blocks render in priority order (structural -> mixed -> status-edit
-> note-only) matching the HTML render so an engineer who has both
windows open sees the same prioritisation. Parser-skip block groups
by reason and emits sample row numbers so a recurring skip pattern
is easy to spot (e.g. "invalid status [4x; rows 5, 12, 18, +1 more]").

Caller-facing sample caps configurable per tier / per adjacent list
/ per skip reason; extras collapse to "...and N more". Determinism
verified by test (same input -> byte-identical text).

### followup-digest-text-html-bundle-i18n-multi-locale

Roll N locale bundles into a single call so a household with
multiple-language caregivers gets one digest per locale from a
single FollowupDigestInput. Today callers loop
`localiseFollowupDigestBundle` per locale and pay for the underlying
digest construction (row selection, null short-circuit, stat
computation) N times.

Output is the per-locale bundle array PLUS a byLocale lookup map
and a coverage rollup with telemetry the QA pipeline can read:
- requestedCount / renderedCount (mismatch surfaces malformed
  locales that refused to render — defensive guard since
  i18n module's null short-circuit is digest-level, not locale-
  level, today).
- locales (deduplicated, in input order).
- noopLocales (locales whose output was character-identical to the
  English baseline — flags empty / placeholder-only locale tables
  for translator follow-up). This is the QA signal: "the ja-JP
  bundle has no entries; it shipped the English text verbatim;
  please review."

Null short-circuit is GLOBAL: when the underlying digest is null
(silent week), the entire multi-locale call returns null so no
caregiver gets an empty pulse in their own language. Duplicate
locale ids dedupe last-wins so household-specific overrides beat
global defaults; input encounter order is preserved for the first
appearance of each locale so output is deterministic.

### refusal-reason-suggest-i18n-rollup-html

HTML render of `RefusalReasonI18nRollupResult` for the patient
adjudication queue UI. Sibling to dose-export-csv-import-roundtrip-
validator-html — same chip palette, same table layout, same font
stack — so the queue feels visually consistent across both modules.

Structural decisions:
- Suggestions GROUPED BY SOURCE in priority order (npo-window ->
  prescriber-pause -> out-of-supply -> sleeping-window ->
  recent-pattern). Each source maps to its own chip colour borrowed
  from followup-digest-html so a fast scan tells the patient which
  rule fired.
- Each row carries the localised explanation, the locale id (with
  a FALLBACK badge when the i18n layer used the English fallback),
  the suggested reason code, and accept / reject controls bound to
  the dose id.
- Doses with no suggestion are dropped — the suggester's "nothing
  fired" entries aren't actionable in the queue.
- Coverage strip at the top mirrors the rollup's telemetry
  (suggested/dose count, top source, fallback count, missing
  placeholders) so the patient sees the queue's health at a glance.
  Missing placeholders are explicitly surfaced because they're the
  QA signal that a locale bundle has bad keys.

Per-source row cap (default 25), per-source filter (drilldown
view), and `renderRefusalReasonI18nRollupTableOnly` variant for
embedding inside an existing adjudication container. `shownBySource`
record lets the caller render per-source pagination without re-counting.

Header underline color is `#6d28d9` (purple) to distinguish this
panel from validator-html's `#0f766e` (teal) at a glance when both
are open in adjacent panels.

### prescriber-contact-card-emergency-card-pdf-two-up

Landscape two-up variant of the emergency card PDF. Two cards
side-by-side on a single sheet, cutting paper use in half. Physical
use case is the cardiology / oncology ED-binder cover: patient
walks in with a 4-card mini-deck on two sheets instead of a 4-page
packet, with a scissor-cut gutter so the cards split into individual
wallet inserts.

Page geometry:
- Same paper (Letter / A4) rotated to LANDSCAPE.
- Vertical centerline divides into left and right halves with a
  configurable gutter (default 18pt / ~0.25") for the scissor cut.
- Each half preserves the single-up card's block ordering and
  visual hierarchy. Font sizes step down a notch (hero 36pt vs
  single-up 48pt, name 16pt vs 18pt, QR 180pt vs 240pt) to fit the
  narrower slot without losing the design.

Crucially we re-do the block math for half-page coordinates rather
than calling the single-up builder twice and offsetting blocks —
the single-up coordinates are page-relative, not half-page-relative,
and naive translation would put the hero phone at the wrong y. The
two layouts share visual language but compute their own positions.

Odd-count terminal page renders the left slot with right=null
(`rightSlotEmpty=true`) so the binder doesn't end on a blank
half-card. Multi-page helper `buildEmergencyCardPdfTwoUpPages`
pairs cards 1+2, 3+4 in input order; page count = ceil(n / 2).
