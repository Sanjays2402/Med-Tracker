## Surprises and lessons

### 1. `TimeBucket` name collision

`packages/utils/src/index.ts` re-exports everything with
`export *`. The first commit shipped `regimen-summary.ts`
exporting a 5-value `TimeBucket` type, which collided with
`pill-burden.ts`'s pre-existing 4-value `TimeBucket`. Tests passed
file-by-file but the full `tsc -p tsconfig.json` flagged TS2308
on `index.ts`. The two types are semantically different (different
windows, different applications), so collapsing them would have
been wrong — renamed `regimen-summary`'s export to
`RegimenTimeBucket` instead. Future modules should adopt
module-prefixed names for any "time of day" / "summary options"
types from the start; this is now the *third* such collision
risk (the others were `BpSummaryOptions` / `GlucoseSummaryOptions`
in tick 2).

### 2. Dedup-by-bucket has boundary bugs

First draft of `medication-history-import` used a
`Math.round(ms / toleranceMs) * toleranceMs` key to map close
timestamps to the same bucket. Failed three tests because rows
at e.g. T=4min and T=6min round to different buckets (0 and
10) even though they're 2 minutes apart, which is well inside
the 5-minute tolerance. Switched to a per-medication list of
`{ ms, index }` with an `abs(diff) <= tolerance` linear scan.
Slightly slower (O(rows-per-med)) but exact at the boundary;
for human-scale histories the cost is invisible. Lesson: when
"within N of an existing value" matters, don't bucket — search.

### 3. Self-introduced test typo

The first `vacation-overrides.test.ts` regimen-wide shift case
had a malformed `every()` predicate (chained ternary inside an
equality) that the LSP caught immediately. Fixed by deleting
the redundant line; the next two assertions already covered the
intent. The LSP diagnostics from `write_file` are worth reading.

### 4. Replace-times validation regex

`isValidOverride` initially accepted `'25:00'` because
`/^\d{2}:\d{2}$/` matches digit shape but doesn't constrain hour
to 0-23. Tightened to `(\d{2}):(\d{2})` + numeric bounds. Same
trap would have existed in `pharmacy-hours.ts` if I'd reused that
regex; future HH:MM validation helpers should live in
`packages/utils/src/date.ts` once we have a real one.

## What to do differently next tick

- Pre-flight a `tsc --noEmit` on `index.ts` AFTER adding each
  re-export to catch name collisions in the same edit cycle
  instead of at the batch gate. Cheap (the package compiles
  in a few seconds) and avoids the fixup-commit pattern.
- Keep watching the `<Module>SummaryOptions` / `<Module>TimeBucket`
  naming conventions; the rule is: if a name could plausibly be
  defined by two modules, prefix it with the module.
- For utilities that take a "window/tolerance" parameter, write
  the dedup logic as a linear `find within tolerance` scan unless
  there's a measured performance reason to bucket.
